## 一、项目概述（Project Overview）

### 1.1 这个系统做什么

**OpenClaw Kids Daily Education Push** 是一套运行在本地 Windows PC 上的自动化教育内容推送系统。它的核心目标很简单：

- **每天早上 06:30**，自动向家长群推送一条「每日科普」知识卡片
- **每天晚上 21:00**，自动向家长群推送一条「每日小古文」学习内容（含原文、出处、图片、学习任务）

推送目标支持三大主流办公协作平台的群机器人：

| 平台 | 文字推送 | 本地图片推送 | 推荐程度 |
|------|----------|--------------|----------|
| **企业微信 (WeChat Work)** | 支持 | **支持 base64 直传** | 首选 |
| **飞书 (Feishu/Lark)** | 支持 | 不支持（需 tenant_access_token 上传） | 次选 |
| **钉钉 (DingTalk)** | 支持 | 不支持 | 备选 |

### 1.2 为什么叫 OpenClaw

OpenClaw 寓意「开放的爪子」—— 像孩子一样好奇地探索世界。Kids Skills 则代表这是一个面向儿童能力培养的内容体系。

### 1.3 推送内容来源

- **科普内容**：来自预置的 `knowledge_base.yaml` 知识库，按星期循环不同学科（周一科技前沿、周二物理世界、周三化学奥秘、周四自然探秘、周五天下大事、周六英语趣谈、周日惊喜知识）
- **小古文内容**：来自用户已有的百度网盘资源，共 282 集，涵盖《论语》《史记》《三字经》《世说新语》等经典出处，每集配有音频（MP3）和三张图片（大卡、小卡、长文稿）

## 三、架构决策：为什么选择 Windows 计划任务而非 GitHub Actions

### 3.1 最初的设想

很多开发者第一反应是：「这种定时任务，用 GitHub Actions 的 `schedule` 触发器不就行了？免费、稳定、云端运行，不需要自己的电脑一直开着。」

这个想法在纯文字推送的场景下确实成立。但我们的项目有一个**致命约束**：

> **小古文推送需要附带本地图片文件**（大卡、长文稿），而这些图片存储在用户的 Windows D 盘上。

### 3.2 GitHub Actions 的不可逾越的障碍

GitHub Actions 运行在微软 Azure 云端的虚拟机上，它的文件系统与用户的本地 PC 完全隔离：

```
GitHub Actions Runner (云端 Ubuntu/Windows 虚拟机)
    │
    ├── 可以访问 GitHub 仓库内的文件
    ├── 可以访问互联网（HTTP API）
    └── ❌ 无法访问 D:\Download\百度网盘下载\图片\小古文（1~240）\
```

如果坚持使用 GitHub Actions，唯一的 workaround 是：

1. 把 282 张图片全部上传到某个图床（阿里云 OSS、腾讯云 COS、GitHub Pages 等）
2. 在索引中存储 URL 而非本地路径
3. 推送时引用远程 URL

但这个方案有多个问题：

- **成本**：图床存储和流量需要付费
- **隐私**：儿童教育内容图片上传到公网存在合规风险
- **复杂度**：需要额外维护一套图床上传/同步机制
- **失败率**：远程图片 URL 可能因网络问题加载失败

### 3.3 企业微信的 base64 图片 API

企业微信（WeChat Work）的群机器人 Webhook 提供了一条「捷径」：

```json
{
  "msgtype": "image",
  "image": {
    "base64": "BASE64_ENCODED_IMAGE_DATA",
    "md5": "IMAGE_MD5_HASH"
  }
}
```

这意味着：**只要能在本地读取到图片文件的二进制内容，就可以直接通过 HTTP POST 发送到企业微信群，无需任何第三方图床**。

而这一能力，只有在**本地运行**的脚本才能实现。

### 3.4 最终决策

| 维度 | GitHub Actions | Windows 计划任务 |
|------|----------------|------------------|
| 访问本地 D 盘图片 | 不可能 | 直接访问 |
| 企业微信 base64 发图 | 不可能 | 完美支持 |
| 电脑关机时的行为 | 云端继续运行 | 任务跳过（需保持开机） |
| 配置复杂度 | 低（YAML 即可） | 中（需 PowerShell 脚本） |
| 调试便利性 | 低（需 push 后等待触发） | 高（本地直接运行测试） |
| 网络依赖 | 仅依赖公网 | 依赖公网 + 本地文件系统 |

**结论**：选择 **Windows 计划任务（Task Scheduler）** 作为调度引擎，配合本地 Python 脚本执行推送。用户的 PC 需要保持开机状态（或设置自动唤醒），但这对于一台日常使用的家庭电脑来说完全可以接受。

## 四、逐步构建过程（Step-by-Step Build Process）

### 4.1 第一步：分析现有代码

#### 4.1.1 科普生成器 `daily-science-kids/scripts/generate.py`

这个脚本的核心逻辑：

1. **查找知识库**：按优先级搜索 `knowledge_base.yaml`（当前目录 → 上级目录 → 用户主目录）
2. **确定今日学科**：根据星期几映射到学科（周一 tech、周二 physics ...）
3. **去重选择**：加载 `.history.json`，排除最近 30 天已推送的知识点，从中随机选择一条
4. **生成文案**：根据 `--grade` 参数生成小学版或初中版文案
5. **记录历史**：将本次推送写入 `.history.json`，保留最近 90 天记录

**关键设计**：

- 使用 `yaml.safe_load()` 解析知识库
- 历史记录使用 JSON 格式，便于人工查看和调试
- 支持 `--preview` 模式，生成文案但不记录历史（用于测试）

#### 4.1.2 小古文生成器 `daily-guguwen/scripts/generate.py`

这个脚本比科普生成器更复杂，因为它需要处理本地文件路径和图片：

1. **加载索引**：从 `index.json` 读取 282 条记录
2. **选择条目**：支持三种模式
   - `--day today`：按日期选择（默认按一年中的第几天循环）
   - `--number 42`：按编号精确选择
   - `--random`：随机选择（优先排除最近 30 条）
3. **生成 Markdown**：拼接标题、出处、原文、图片引用、音频链接、学习任务
4. **记录历史**：写入 `.history.json`

**关键设计**：

- 索引中的 `images` 字段存储了三个键：`big_card`、`small_card`、`long_text`
- 图片路径是绝对路径（如 `D:\Download\...`），因为索引是在用户本地机器上生成的
- 音频路径使用 `file://` 协议，方便在支持本地链接的客户端中直接播放

### 4.2 第二步：编写统一入口脚本 `scripts/local_push.py`

这是整个系统的「心脏」—— 一个同时支持早上科普和晚上小古文的统一推送入口。

#### 4.2.1 设计目标

- 一个脚本，两种模式：`morning` / `evening`
- 支持三种平台：`wechat` / `feishu` / `dingtalk`
- 企业微信模式下，小古文推送可附带本地图片
- 所有配置通过命令行参数或环境变量传入，便于计划任务调用

#### 4.2.2 核心函数解析

**`push_text()` —— 文字消息推送**

三个平台的 payload 结构各不相同：

- **企业微信**：`{"msgtype": "markdown", "markdown": {"content": "..."}}`
- **飞书**：`{"msg_type": "interactive", "card": {"header": {...}, "elements": [...]}}`
- **钉钉**：`{"msgtype": "markdown", "markdown": {"title": "...", "text": "..."}}`

企业微信有 4096 字节的 Markdown 内容限制，超过后自动截断并附加提示。

**`push_image_wechat()` —— 企业微信图片推送**

```python
def push_image_wechat(webhook_url: str, image_path: Path):
    data = image_path.read_bytes()              # 读取本地文件二进制
    b64 = base64.b64encode(data).decode("ascii") # Base64 编码
    md5 = hashlib.md5(data).hexdigest()          # MD5 哈希
    payload = {"msgtype": "image", "image": {"base64": b64, "md5": md5}}
    requests.post(webhook_url, json=payload)
```

这是整个系统中最关键的函数。它直接读取本地图片文件，将其转换为 base64 字符串，连同 MD5 校验值一起发送给企业微信服务器。企业微信收到后，会在群聊中直接显示这张图片。

**`_read_image_bytes()` —— 图片压缩兜底**

企业微信对单张图片有 2MB 大小限制。如果图片超过限制，脚本会自动尝试压缩：

1. 先尝试降低 JPEG 质量（85% → 70% → 55% → 40% → 25%）
2. 如果仍超限，逐步降低分辨率（每次减半，最低 400×400）
3. 如果 Pillow 不可用或压缩失败，回退到发送原图（企业微信会返回错误提示）

**`generate_guguwen()` —— 小古文内容生成（增强版）**

与原始的 `daily-guguwen/scripts/generate.py` 不同，`local_push.py` 中的版本做了以下增强：

1. 调用 `generate.py` 时传入 `--base-date 2026-05-15`，使今天成为第 1 天（详见第五节）
2. 从 `index.json` 中读取今日条目的图片路径
3. 检查图片文件是否真实存在（`Path(p).exists()`）
4. 返回结构化数据：`{"text": "...", "images": [Path, ...]}`

**`run_morning()` / `run_evening()` —— 主流程**

两个函数分别处理早上和晚上的推送流程：

- **Morning**：生成科普文案 → 发送文字消息 → 完成
- **Evening**：生成小古文文案 → 发送文字消息 → 如果是企业微信则逐张发送图片 → 完成

都使用 `tempfile.NamedTemporaryFile` 创建临时文件，并在 `finally` 块中清理，避免磁盘垃圾积累。

#### 4.2.3 命令行接口

```bash
# 早上科普
python scripts/local_push.py morning --webhook "https://qyapi.weixin.qq.com/..." --type wechat

# 晚上小古文（含图片）
python scripts/local_push.py evening --webhook "https://qyapi.weixin.qq.com/..." --type wechat

# 使用环境变量（计划任务推荐方式）
set WEBHOOK_URL=https://qyapi.weixin.qq.com/...
set WEBHOOK_TYPE=wechat
python scripts/local_push.py morning
```

### 4.3 第三步：编写 PowerShell 计划任务安装脚本 `scripts/setup_windows_task.ps1`

#### 4.3.1 为什么用 PowerShell

Windows 计划任务的图形界面操作繁琐，且不利于版本控制和重复部署。使用 PowerShell 脚本可以：

- 一键创建/更新任务
- 自动查找 Python 解释器路径
- 将 Webhook URL 和类型直接嵌入任务参数
- 支持以管理员身份运行（`#Requires -RunAsAdministrator`）

#### 4.3.2 脚本核心逻辑

**自动查找 Python**

```powershell
$candidates = @("python", "python3", "py")
foreach ($c in $candidates) {
    $found = (Get-Command $c -ErrorAction SilentlyContinue).Source
    if ($found) { $PythonPath = $found; break }
}
```

按优先级尝试 `python`、`python3`、`py`，找到第一个可用的即停止。

**构建任务参数**

```powershell
$ArgMorning = "`"" + $ScriptPath + "`"" morning --webhook `"" + $WebhookUrl + "`"" --type `"" + $WebhookType + "`""
```

这里使用了复杂的引号转义，确保路径中包含空格时也能正确解析。最终生成的参数字符串形如：

```
"C:\Users\Admin\openclaw-kids-skills\scripts\local_push.py" morning --webhook "https://..." --type "wechat"
```

**任务触发器**

```powershell
$TriggerMorning = New-ScheduledTaskTrigger -Daily -At "06:30"
$TriggerEvening = New-ScheduledTaskTrigger -Daily -At "21:00"
```

**任务设置（关键）**

```powershell
$Settings = New-ScheduledTaskSettingsSet `
    -WakeToRun `                          # 如果电脑休眠，唤醒后运行
    -StartWhenAvailable `                 # 如果错过时间，尽快补运行
    -MultipleInstances IgnoreNew `        # 不重复运行新实例
    -RestartCount 2 `                     # 失败后重试 2 次
    -RestartInterval (New-TimeSpan -Minutes 5)
```

**Principal（运行身份）**

```powershell
$Principal = New-ScheduledTaskPrincipal -UserId "$env:USERDOMAIN\$env:USERNAME" `
    -LogonType S4U `
    -RunLevel Limited
```

`-LogonType S4U`（Service for User）是**最关键的配置**。它允许任务在以下情况下运行：

- 用户已登录
- 用户锁定屏幕
- 用户注销后（如果设置了持久化，但 S4U 本身不支持完全注销后运行）

相比传统的 `Password` 登录类型，S4U 不需要在任务中存储明文密码，安全性更高。

#### 4.3.3 脚本使用方式

```powershell
# 企业微信
.\scripts\setup_windows_task.ps1 -WebhookUrl "https://qyapi.weixin.qq.com/cgi-bin/webhook/send?key=xxx" -WebhookType "wechat"

# 飞书
.\scripts\setup_windows_task.ps1 -WebhookUrl "https://open.feishu.cn/open-apis/bot/v2/hook/xxx" -WebhookType "feishu"
```

### 4.4 第四步：编写 `.env.example` 配置模板

虽然计划任务已经将 Webhook URL 嵌入到了任务参数中，但提供一个 `.env.example` 模板仍然有价值：

1. **本地测试**：在命令行手动运行脚本时，可以通过 `.env` 文件加载配置
2. **文档作用**：注释中详细说明了三个平台的 URL 格式和注意事项
3. **扩展预留**：如果未来需要增加更多配置项（如图片压缩阈值、重试次数），可以统一放在 `.env` 中


### 4.5 第五步：飞书 Webhook 集成原理

飞书的群机器人通过自定义 Webhook 接收消息：

1. 在飞书群设置中添加「自定义机器人」
2. 复制生成的 Webhook URL（形如 `https://open.feishu.cn/open-apis/bot/v2/hook/xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx`）
3. 向该 URL 发送 POST 请求，Body 为 JSON

**飞书消息卡片结构**：

```json
{
  "msg_type": "interactive",
  "card": {
    "header": {
      "title": {"tag": "plain_text", "content": "每日小古文 · 第1天"},
      "template": "blue"
    },
    "elements": [
      {"tag": "div", "text": {"tag": "lark_md", "content": "**出处**：论语\n**原文**：不怨天，不尤人"}}
    ]
  }
}
```

飞书的卡片消息比纯文本更美观，支持颜色主题（`blue`、`green`、`orange` 等），但不支持直接发送本地图片。如需发图，必须先调用「上传图片」API 获取 `image_key`，再引用该 key 发送消息，流程复杂得多。

### 4.6 第六步：企业微信图片推送原理

企业微信的群机器人 Webhook 支持三种消息类型：`text`、`markdown`、`image`。其中 `image` 类型的特殊之处在于：

```json
{
  "msgtype": "image",
  "image": {
    "base64": "iVBORw0KGgoAAAANSUhEUgAA...",
    "md5": "5d2f5c8a5b6e7d8f9a0b1c2d3e4f5a6b"
  }
}
```

- `base64`：图片文件的 Base64 编码字符串
- `md5`：图片原始二进制内容的 MD5 哈希值（小写 32 位十六进制）

**为什么需要 MD5**：企业微信用 MD5 做图片去重和校验。如果同一个 MD5 在短时间内重复发送，可能会被拦截或合并。

**大小限制**：单张图片不超过 2MB。超过后企业微信会返回 `errcode=40008`（无效消息）。

**优势**：

- 无需任何第三方图床或 CDN
- 图片直接存储在企业微信服务器上，稳定性由腾讯保障
- 群成员看到的图片与本地文件完全一致，无压缩失真（除非我们主动压缩）

## 五、日期计数修正（Day Counting Fix）

### 5.1 原始逻辑的问题

在早期的 `daily-guguwen/scripts/generate.py` 中，日期选择逻辑是：

```python
day_of_year = target.timetuple().tm_yday  # 一年中的第几天，1月1日=1
idx = (day_of_year - 1) % len(index)       # 取模循环
```

这意味着：

- 1 月 1 日 → 第 1 集
- 1 月 2 日 → 第 2 集
- ...
- 5 月 15 日（2026 年是第 135 天）→ **第 135 集**

如果用户在 2026-05-15 首次启动系统，会直接跳到第 135 集《论语·子罕》，错过了前面 134 集的内容。这对于一个「每天循序渐进」的学习计划来说是不可接受的。

### 5.2 修正方案：`--base-date` 参数

我们在 `generate.py` 中增加了 `--base-date` 参数：

```python
def select_by_day(index, year=None, month=None, day=None, base_date=None):
    now = datetime.now()
    target = datetime(year or now.year, month or now.month, day or now.day)

    if base_date:
        delta = (target - base_date).days  # 从基准日期起经过的天数
        idx = delta % len(index)            # 今天 = 第 delta 天
    else:
        day_of_year = target.timetuple().tm_yday
        idx = (day_of_year - 1) % len(index)
    return index[idx]
```

当调用时传入 `--base-date 2026-05-15`：

- 2026-05-15 → delta = 0 → idx = 0 → **第 1 集**
- 2026-05-16 → delta = 1 → idx = 1 → **第 2 集**
- ...

这样，无论用户在一年中的哪一天启动系统，都能从第 1 集开始顺序播放。

### 5.3 在 `local_push.py` 中的应用

```python
subprocess.run(
    [sys.executable, str(GUGUWEN_SCRIPT), "--index", str(GUGUWEN_INDEX),
     "--day", "today", "--base-date", "2026-05-15", "-o", str(output_path)],
    check=True,
    cwd=str(REPO_ROOT),
)
```

`--base-date` 被硬编码为 `2026-05-15`，因为这就是系统首次投入运行的日期。如果未来需要重新开始（比如 282 集全部播完后想再来一轮），只需修改这个日期即可。

## 六、踩坑记录与解决方案（Problems & Solutions）

### 6.1 问题一：GitHub Actions 无法访问本地文件

**现象**：最初尝试用 GitHub Actions 的 `schedule` 触发器定时运行脚本，但推送小古文时找不到图片文件。

**根因**：GitHub Actions Runner 运行在云端虚拟机，没有挂载用户的 D 盘。

**解决**：放弃 GitHub Actions，改用 Windows 计划任务。在本地 PC 上运行脚本，直接读取 `D:\Download\...` 路径下的文件。

**经验教训**：

- 任何涉及本地文件（尤其是大体积、隐私敏感的文件）的自动化任务，都应优先考虑本地运行方案
- 云端方案适合「无状态」任务（纯 API 调用、纯计算），本地方案适合「有状态」任务（访问本地数据库、文件、硬件）

### 6.2 问题二：PowerShell 脚本 LF 换行导致解析失败

**现象**：在 VS Code 中编写 `setup_windows_task.ps1` 后，在 Windows PowerShell 5.1 中执行时报错：

```
无法将参数绑定到参数“Command”，因为该参数为空字符串。
```

**根因**：VS Code 默认使用 LF（`\n`）作为换行符，但 Windows PowerShell 5.1 要求脚本文件使用 CRLF（`\r\n`）。当脚本以 LF 保存时，PowerShell 解析器在某些情况下会将整行内容错误地合并，导致参数绑定失败。

**解决**：使用 .NET 的 `IO.File.WriteAllText` 方法，显式以 CRLF 换行符写入文件：

```powershell
[System.IO.File]::WriteAllText($FilePath, $Content, [System.Text.Encoding]::UTF8)
```

或者在 VS Code 右下角点击「CRLF」/「LF」切换换行符模式。

**经验教训**：

- Windows PowerShell 5.1（系统自带）对 LF 的支持不如 PowerShell 7.x（跨平台版）
- 涉及 PowerShell 脚本的文件操作，务必确认换行符格式
- 如果使用 Git，可以在 `.gitattributes` 中设置 `*.ps1 text eol=crlf`

### 6.3 问题三：计划任务参数缺少 Webhook URL

**现象**：手动运行 `local_push.py` 成功，但计划任务触发时脚本报错「未设置 WEBHOOK_URL」。

**根因**：计划任务的「操作」配置中，`Argument` 字段没有包含 `--webhook` 参数。脚本既没从命令行参数获取到 URL，也没从环境变量获取到，于是报错退出。

**解决**：在 `setup_windows_task.ps1` 中，将 `--webhook` 和 `--type` 直接嵌入到任务参数中：

```powershell
$ArgMorning = "`"" + $ScriptPath + "`"" morning --webhook `"" + $WebhookUrl + "`"" --type `"" + $WebhookType + "`""
```

这样，即使用户没有设置环境变量，计划任务也能正确传递配置。

**经验教训**：

- 计划任务的环境变量与交互式 PowerShell 会话的环境变量是隔离的
- 最可靠的方式是将所有必要参数直接写入任务的 `Argument` 字段
- 如果确实需要环境变量，应在任务的「条件」或「操作」中显式设置

### 6.4 问题四：飞书返回「token invalid」—— URL 被截断

**现象**：复制飞书 Webhook URL 到 PowerShell 参数时，脚本执行后飞书返回错误：

```json
{"code": 99991663, "msg": "token invalid"}
```

**根因**：飞书 Webhook URL 很长（约 80 个字符），在终端中复制时，Windows 终端的自动换行使 URL 被断成了两行。用户只复制了第一行，导致 URL 不完整（缺少末尾的 token 部分）。

**解决**：创建了一个 `setup_now.ps1` 一键配置脚本，将 Webhook URL 直接写在脚本文件中：

```powershell
# 一键配置 OpenClaw 定时推送（飞书版）
.\scripts\setup_windows_task.ps1 `
    -WebhookUrl "https://open.feishu.cn/open-apis/bot/v2/hook/af19931c-fafa-4c55-af9c-a5dff9f9530f" `
    -WebhookType "feishu"
```

用户只需运行 `.\setup_now.ps1`，无需手动复制粘贴长 URL。

**经验教训**：

- 长 URL 在终端中显示时容易因换行而被截断
- 对于一次性配置脚本，将敏感信息直接写在脚本中比命令行传参更可靠（脚本文件本身可以受文件系统权限保护）
- 生产环境中应考虑将 Webhook URL 存储在 Windows 凭据管理器或加密文件中

### 6.5 问题五：Windows 终端 UTF-8 编码问题（GBK 无法编码 Emoji）

**现象**：运行脚本时，终端输出中出现乱码或报错：

```
UnicodeEncodeError: 'gbk' codec can't encode character '\U0001f4dc' in position 45: illegal multibyte sequence
```

**根因**：Windows 中文系统的默认代码页是 GBK（代码页 936），而脚本中使用了 Emoji（如 📜、📚、📝）。GBK 编码不支持这些 Unicode 字符，导致编码错误。

**解决**：

1. **脚本层面**：Python 脚本头部已声明 `# -*- coding: utf-8 -*-`，且所有文件读写操作都显式指定了 `encoding="utf-8"`。脚本本身的逻辑是正确的。

2. **终端层面**：在运行脚本前，执行以下命令切换终端代码页为 UTF-8：

```cmd
chcp 65001
```

或者在 PowerShell 中：

```powershell
[Console]::OutputEncoding = [System.Text.Encoding]::UTF8
```

3. **长期方案**：在系统环境变量中添加 `PYTHONIOENCODING=utf-8`，使 Python 强制使用 UTF-8 编码标准输出。

**经验教训**：

- Windows 终端的编码问题是一个历史遗留痛点，GBK 与 UTF-8 的混用经常导致乱码
- 脚本中应始终显式指定 `encoding="utf-8"`，不要依赖系统默认编码
- Emoji 在 Markdown 消息中能增强视觉吸引力，但在日志输出中可以考虑用纯文本替代，避免终端兼容性问题

## 七、各平台绑定指南（Platform Binding Guides）

### 7.1 企业微信（WeChat Work）—— 强烈推荐

**适用场景**：需要发送本地图片（小古文大卡、长文稿）

**绑定步骤**：

1. 打开企业微信，进入目标群聊
2. 点击右上角「...」→「群机器人」→「添加机器人」
3. 输入机器人名称（如「OpenClaw 小助手」），点击「确定」
4. 复制 Webhook 地址（形如 `https://qyapi.weixin.qq.com/cgi-bin/webhook/send?key=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx`）
5. 运行安装脚本：

```powershell
.\scripts\setup_windows_task.ps1 `
    -WebhookUrl "https://qyapi.weixin.qq.com/cgi-bin/webhook/send?key=你的key" `
    -WebhookType "wechat"
```

**特点**：

- 支持 Markdown 文字消息
- **支持 base64 编码的本地图片直传**（单张 < 2MB）
- 支持 @所有人（需在 payload 中加 `"mentioned_list": ["@all"]`）
- 免费，无调用次数限制（但频繁调用可能触发风控）

### 7.2 飞书（Feishu/Lark）

**适用场景**：团队已在使用飞书，且对图片需求不高（或愿意手动查看本地图片）

**绑定步骤**：

1. 打开飞书，进入目标群聊
2. 点击右上角「...」→「设置」→「群机器人」→「添加机器人」→「自定义机器人」
3. 输入机器人名称，点击「确定」
4. 复制 Webhook 地址（形如 `https://open.feishu.cn/open-apis/bot/v2/hook/xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx`）
5. 运行安装脚本：

```powershell
.\scripts\setup_windows_task.ps1 `
    -WebhookUrl "https://open.feishu.cn/open-apis/bot/v2/hook/你的token" `
    -WebhookType "feishu"
```

**特点**：

- 支持交互式卡片消息（比纯 Markdown 更美观）
- 不支持直接发送本地图片（需先调用图片上传 API 获取 `image_key`）
- 卡片消息支持颜色主题、按钮、分割线等富媒体元素
- 免费版有调用频率限制（具体以飞书官方文档为准）

### 7.3 钉钉（DingTalk）

**适用场景**：团队已在使用钉钉

**绑定步骤**：

1. 打开钉钉，进入目标群聊
2. 点击右上角「群设置」→「智能群助手」→「添加机器人」→「自定义」
3. 输入机器人名称，选择安全设置（如「加签」或「IP 地址」）
4. 复制 Webhook 地址（形如 `https://oapi.dingtalk.com/robot/send?access_token=xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx`）
5. 运行安装脚本：

```powershell
.\scripts\setup_windows_task.ps1 `
    -WebhookUrl "https://oapi.dingtalk.com/robot/send?access_token=你的token" `
    -WebhookType "dingtalk"
```

**特点**：

- 支持 Markdown 文字消息
- 不支持直接发送本地图片
- 安全设置较严格（可能需要处理加签逻辑，当前脚本暂未实现）
- 免费版有调用频率限制

## 八、防坑指南（Anti-Pitfall Guide）

### 8.1 管理员权限

创建 Windows 计划任务需要管理员权限。运行 PowerShell 脚本时：

- 右键点击 PowerShell →「以管理员身份运行」
- 脚本头部已包含 `#Requires -RunAsAdministrator`，非管理员运行时会自动提示

### 8.2 Webhook URL 的复制

Webhook URL 通常很长（60-100 个字符），复制时务必：

- 在浏览器或企业微信客户端中，使用「复制链接」按钮
- 不要手动从终端输出中复制（终端换行可能导致截断）
- 粘贴到脚本中后，检查 URL 是否以 `http` 开头、以 token/key 结尾，中间没有空格或换行

### 8.3 电脑保持开机

Windows 计划任务依赖本地 PC 运行。请确保：

- PC 在 06:30 和 21:00 处于开机状态
- 如果 PC 进入睡眠/休眠，任务可能无法触发（虽然设置了 `-WakeToRun`，但部分硬件不支持唤醒）
- 建议将电源计划设置为「从不进入睡眠」（控制面板 → 电源选项）

### 8.4 修改 Webhook 后的重新配置

如果更换了群机器人（如 Webhook URL 变更），需要：

1. 修改 `setup_now.ps1` 中的 Webhook URL
2. 重新运行 `.\setup_now.ps1`
3. 脚本会自动删除旧任务并创建新任务

### 8.5 图片推送的平台限制

**只有企业微信支持直接发送本地图片**。如果当前配置的是飞书或钉钉：

- 脚本会正常发送文字消息
- 图片会被跳过，并在日志中提示：「feishu 暂不支持直接发送本地图片」
- 家长可以在本地文件夹中查看图片，或改用企业微信

### 8.6 临时文件清理

`local_push.py` 使用 `tempfile.NamedTemporaryFile` 创建临时 Markdown 文件。正常情况下，脚本会在 `finally` 块中自动删除这些文件。如果脚本异常退出（如强制终止进程），临时文件可能残留。

清理方法：

```powershell
# 删除所有临时 Markdown 文件
Remove-Item $env:TEMP\tmp*.md -ErrorAction SilentlyContinue
```

### 8.7 索引文件同步

如果用户新增了小古文资源（如从百度网盘下载了第 283 集），需要重新运行索引器：

```bash
python daily-guguwen/scripts/indexer.py `
    --audio "D:\Download\百度网盘下载\每日小古文音频" `
    --images "D:\Download\百度网盘下载\图片\小古文（1~240）" `
    --output daily-guguwen/index.json
```

然后重新运行 `setup_now.ps1` 以确保计划任务引用最新的索引。
